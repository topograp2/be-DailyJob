### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 포스트에 대한 댓글 관련 조회 책임을 `PostRepository`에 넣지 않고 `CommentRepository`에 분리했다.
- 그렇게 판단한 이유: 게시글 상세 화면에서 댓글 개수나 댓글 목록이 필요하더라도 댓글 데이터의 소유 주체는 `comments` 테이블이다. 따라서 댓글 개수 조회나 댓글 목록 조회는 `CommentRepository`가 담당하는 것이 책임 분리에 더 적절하다고 판단했다.
- 고려했지만 선택하지 않은 대안: `PostRepository`의 게시글 상세 조회 쿼리에 댓글 개수나 댓글 목록 조회 로직을 함께 넣는 방법도 가능했다. 하지만 그렇게 하면 `PostRepository`가 댓글 조회 방식까지 알아야 해서 역할이 커지고, 이후 댓글 정렬 방식이나 응답 형식이 바뀔 때 게시글 조회 코드까지 함께 수정해야 하는 문제가 생길 수 있다고 봤다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 댓글 조회 로직이 `CommentRepository`에 모이면서 각 repository의 책임이 더 명확해졌다. 이후 댓글 전체 조회 같은 기능이 추가되더라도 변경 지점이 댓글 패키지 내부로 제한된다. 또한 `PostService`나 `CommentService`에서는 필요한 데이터를 조합하는 역할에 집중할 수 있어 테스트할 단위도 더 분명해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 댓글 목록을 화면에 보여주려면 `CommentRepository`에서 댓글을 조회하는 것만으로는 부족했다. 화면에는 댓글 내용뿐 아니라 작성자 이름과 프로필 이미지 경로도 필요했기 때문에 이 정보를 어느 계층에서 조합해야 할지 고민하게 되었다.
- 처음 추측한 원인: 처음에는 댓글 조회 쿼리 자체를 어떻게 작성해야 하는지가 핵심 문제라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: `CommentRepository`, `CommentService`, `CommentDetailResponse`, `CommentHtmlCreator`의 역할을 나눠서 생각했다. 특히 repository가 DB 조회만 담당해야 하는지, 아니면 response 생성까지 관여해도 되는지를 기준으로 구조를 검토했다.
- 실제 원인: 실제로 막힌 지점은 SQL 자체보다 계층 간 책임 분리였다. 댓글 데이터는 `CommentRepository`에서 가져오면 되지만, 작성자 정보는 `UserManager`, 프로필 이미지는 `ImageRepository`를 통해 가져와야 했다. 따라서 repository에서 모든 화면 정보를 만들려고 하면 책임이 과하게 커지는 문제가 있었다.
- 해결 방법: `CommentRepository`는 `postId`에 해당하는 최근 댓글 목록만 `List<Comment>`로 반환하도록 했다. 이후 `CommentService`에서 각 댓글의 `userId`로 사용자 정보를 조회하고, 사용자의 `imageId`로 프로필 이미지 경로를 조회한 뒤 `CommentDetailResponse`로 통합시켰다. HTML 조각 생성은 별도의 `CommentHtmlCreator`로 분리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 화면에 필요한 데이터가 여러 도메인에 걸쳐 있을 때, repository에서 한 번에 다 가져오려 하기 전에 각 계층의 책임을 먼저 확인해야 한다. repository는 데이터 조회, service는 도메인 간 조합 등 각 계층 구조가 적절한지 먼저 판단할 것이다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 데이터 모델링 수업 중 이해가 부족하거나 추가로 확인하면 좋겠다고 느낀 내용을 기록해두고 수업 이후에 학습했다. 특히 MySQL 위주로 개발해와서 익숙하지 않았던 Oracle의 `DATETIME`, `TIMESTAMP`, `VARCHAR`, `VARCHAR2` 타입 차이를 확인했고 ERD를 작성하면서 `VARCHAR`와 `VARCHAR2`가 함께 등장했을 때 두 타입의 차이를 그냥 넘기지 않고 찾아봤다.
- 그 접근이 도움이 된 이유: 수업 중 모르는 개념을 즉시 완전히 이해하지 못하더라도 나중에 다시 확인할 수 있도록 기록해두었기 때문에 잊지 않고 학습할 수 있었다. 또한 "Oracle은 MySQL과 다르다" 정도로 넘기지 않고, 실제 DB 타입의 차이를 비교하면서 데이터베이스마다 타입 선택 기준이 달라질 수 있다는 점을 이해할 수 있었다. 항공권 예약처럼 동시성 문제가 중요한 상황에서 DB 모델링으로 문제를 줄일 수 있는 전략도 함께 학습하면서 ERD 작성이 실제 서비스 안정성과 연결된다는 점을 인식하게 되었다.
- 다음에도 반복하고 싶은 점: 수업이나 개발 중 모르는 개념이 나왔을 때 나중에 확인할 것으로 기록해둔 뒤 실제로 찾아보는 습관을 정착시키고 싶다. 특히 익숙한 기술 기준으로만 판단하지 않고, 다른 DB나 환경에서는 같은 개념이 어떻게 다르게 동작하는지 비교해서 학습하는 방식을 반복하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 개발 중 헷갈리는 부분이 있었을 때 어제처럼 생성형 AI에게 질문한 답변을 노션에 옮겨두고 다시 읽어보는 과정을 충분히 하지 못했다. 오늘도 여러 기술적 판단과 고민이 있었지만 기록으로 남기기보다 대화 흐름 속에서 이해하고 넘어가는 경우가 많았다.
- 그렇게 된 원인: 구현해야 할 기능과 수정할 부분이 계속 이어지면서 당장 다음 작업으로 넘어가는 데 집중했던 것 같다. 그 순간에는 이해했다고 느꼈지만, 따로 정리하지 않으면 나중에 같은 문제를 다시 만났을 때 판단 근거를 빠르게 떠올리기 어려울 수 있다.
- 다음에는 다르게 해볼 구체적인 행동: 개발 중 새롭게 알게 된 내용이나 판단이 필요했던 지점이 나오면, 작업이 끝난 뒤 따로 시간을 내서 노션에 정리할 것이다. 특히 "왜 이렇게 했는지", "선택하지 않은 대안은 무엇인지"를 짧게라도 남기고자 노력해야겠다.
