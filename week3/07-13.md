### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 업로드된 이미지 파일 자체는 로컬 파일 시스템의 별도 이미지 저장 폴더에 저장하고, DB에는 저장된 파일의 경로와 파일명만 저장하기로 했다.
  - 저장 파일명은 원본 파일명을 그대로 사용하지 않고 UUID를 이용해 새로 생성하기로 했다.
- 그렇게 판단한 이유:
  - 현재는 AWS S3를 사용할 수 없는 환경이기 때문에 로컬에서 이미지를 저장하고 관리하는 방식이 필요했다.
  - 이미지를 BLOB으로 DB에 직접 저장하면 DB 용량이 커질 뿐 아니라, 유저 정보를 조회할 때마다 실제로 필요하지 않은 이미지 데이터까지 함께 읽어올 수 있다.
  - 특히 유저 데이터는 로그인 후 모든 페이지 상단바에 닉네임을 표시하기 위해 자주 조회되므로 프로필 이미지 같은 무거운 데이터를 같은 테이블에 두면 조회 성능에 악영향을 줄 수 있다고 판단했다.
  - DB에는 가벼운 경로 문자열만 저장하면 유저 조회 시 불필요한 이미지 데이터 로딩을 피할 수 있다.
  - 원본 파일명을 그대로 저장 파일명으로 사용하면 서로 다른 유저가 같은 이름의 파일을 업로드했을 때 파일이 덮어씌워지거나 저장에 실패할 수 있으므로, UUID로 충돌 가능성을 줄이는 방식이 더 안전하다고 판단했다.
- 고려했지만 선택하지 않은 대안:
  - 이미지 데이터를 BLOB으로 DB에 저장하는 방식
    - 이미지까지 한 곳에서 관리할 수 있다는 장점은 있지만, 자주 조회되는 테이블에서 불필요하게 무거운 데이터를 함께 읽게 될 수 있어 선택하지 않았다.
  - 프로필 사진 전용 1:1 테이블을 따로 만드는 방식
    - 유저 테이블에서 BLOB을 분리할 수는 있지만, 프로필 사진 하나를 위해 별도 테이블과 JOIN 구조를 추가하는 것은 현재 요구사항에 비해 복잡하다고 판단했다.
    - 또한, 추후에 Post 테이블도 만들게 되면 그때도 또 1:1 테이블을 만들게 될 텐데 사진이 필요해질 때마다 무분별하게 이미지를 위한 테이블을 생성하는 것은 복잡하다고 판단하였다.
  - 원본 파일명을 그대로 저장 파일명으로 사용하는 방식
    - 구현은 단순하지만 파일명 충돌 위험이 있어 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 이미지 저장 책임을 파일 시스템 저장 로직과 DB 메타데이터 저장 로직으로 나눌 수 있어 구조가 더 명확해졌다.
  - DB에는 이미지 경로와 파일명만 저장하므로 기존 유저 조회 흐름에 큰 부담을 주지 않는다.
  - UUID 기반 파일명 생성으로 파일명 충돌을 방지할 수 있어 업로드 기능의 안정성이 높아졌다.
  - ImageStorageTest, ImageControllerTest를 통해 파일 저장과 업로드 흐름을 따로 확인할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 이미지 업로드를 위해 multipart/form-data를 처리해야 했는데, 기존 RequestParser는 BufferedReader 기반으로 request를 읽고 있었다.
  - text/html, application/x-www-form-urlencoded처럼 전체가 텍스트인 요청에서는 문제가 없었지만, multipart/form-data의 파일 파트는 바이너리 데이터이기 때문에 문자열로 읽으면 데이터가 손상될 수 있었다.
- 처음 추측한 원인:
  - request body를 char 단위로 읽고 있어서 이미지 같은 바이너리 데이터를 정확히 보존하지 못한다고 생각했다.
  - header는 텍스트이므로 기존처럼 BufferedReader로 읽고, body부터 Content-Type에 따라 byte로 읽으면 해결할 수 있을 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - 기존 RequestParser가 request line, header, body를 어떤 순서와 단위로 읽는지 확인했다.
  - multipart/form-data의 실제 요청 예시를 확인하며 구조를 파악하였다.
  - Content-Length만큼 char 배열을 만들어 body를 읽는 기존 방식이 multipart 파일 데이터에도 문제가 없이 그대로 적용되는지 확인했다.
  - header까지만 BufferedReader로 읽고 이후 body를 InputStream에서 byte로 읽는 방식이 가능한지 검토했다.
- 실제 원인:
  - BufferedReader는 readLine으로 header만 읽는 것처럼 보여도 내부 버퍼 크기만큼 미리 데이터를 읽어올 수 있다.
  - 이 과정에서 body의 앞부분까지 BufferedReader 내부 버퍼에 들어가 버리면, 이후 원래 InputStream에서 body를 byte 단위로 읽으려고 해도 이미 일부 데이터가 이미 읽힌 상태가 된다.
  - 따라서 header는 BufferedReader로, body는 InputStream으로 나눠 읽는 방식은 같은 요청 스트림을 안정적으로 처리하기 어렵다.
- 해결 방법:
  - RequestParser가 처음부터 끝까지 BufferedInputStream 기반으로 request를 byte 단위로 읽도록 변경했다.
  - request line과 header처럼 줄 단위 처리가 필요한 부분은 별도의 readLine 메서드를 구현해 byte를 읽다가 줄 끝을 만나면 문자열로 변환하도록 했다.
  - body는 Content-Length만큼 byte 배열로 읽어 보존하고, multipart/form-data인 경우 Content-Type에서 boundary를 파싱한 뒤 boundary 기준으로 각 part를 나누어 처리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 네트워크 스트림을 처리할 때 Reader와 InputStream을 섞어 쓰고 있는지 먼저 확인한다.
  - body에 바이너리 데이터가 포함될 수 있는 요청인지 확인하고, 가능성이 있다면 처음부터 byte 기준으로 파싱하는 구조를 선택한다.
  - BufferedReader, BufferedInputStream처럼 내부 버퍼를 가진 클래스가 실제로 어디까지 데이터를 미리 읽는지 고려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - PR이나 회고를 작성할 때 추상적으로 적은 부분이 있으면 다시 읽고 더 구체적인 표현으로 고치려고 했다.
  - 이전에는 작성한 내용을 크게 검토하지 않고 바로 마무리하는 편이었는데, 오늘은 내가 적은 말이 다른 사람도 이해할 수 있는 언어인지 확인했다.
- 그 접근이 도움이 된 이유:
  - PR 설명이 구체적일수록 변경 의도와 판단 근거를 더 빠르게 이해할 수 있다.
  - 회고도 막연한 감상으로 끝내지 않고, 내가 어떤 문제를 만났고 어떤 기준으로 판단했는지 남길 수 있었다.
  - 글을 다시 검토하는 과정에서 내가 구현한 내용을 한 번 더 정리하게 되어, 코드 변경의 이유와 흐름을 더 명확히 볼 수 있었다.
- 다음에도 반복하고 싶은 점:
  - PR을 올리기 전에 추상적인 표현이 있는지 한 번 더 확인한다.
  - 무엇을, 왜, 어떻게 바꿨는지까지 적는다.
  - 회고와 PR 기록도 나중에 다시 볼 수 있는 나만의 개발 자산이라는 점을 기억하고, 다른 사람이 읽어도 이해 가능한 문장으로 남긴다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 이미지 저장 서비스 로직을 구현할 때 정상적으로 업로드되는 흐름에만 집중했다.
  - body에 file 내용이 없을 때, 이미지 파일 크기가 너무 클 때처럼 서비스가 방어해야 하는 예외 상황을 처음부터 고려하지 못했다.
  - 결국 테스트를 작성하면서 빠진 예외 케이스를 발견했고, 그때서야 검증 로직을 추가하게 되었다.
- 그렇게 된 원인:
  - 프론트엔드에서 잘못된 요청을 막아줄 것이라고 무의식적으로 기대하는 습관이 있다.
  - 하지만 지금은 프론트엔드 없이 혼자 구현하는 상황이고, 실제 협업에서도 백엔드는 프론트엔드가 항상 올바른 요청만 보낸다고 가정하면 안 된다.
  - 기능이 정상적으로 동작하는지에 집중하다 보니, 이 API가 어떤 입력을 허용하고 어떤 입력을 거절해야 하는지 먼저 정리하지 않았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 서비스 로직을 구현하기 전에 정상 케이스, 빈 값, 누락된 값, 너무 큰 값, 잘못된 타입을 생각해본다.
  - 테스트를 작성할 때 성공 테스트를 먼저 끝내더라도, 바로 이어서 실패해야 하는 요청을 최소 2개 이상 작성한다.
  - 프론트엔드가 막아줄 수 있는 검증이라도 백엔드에서 다시 검증해야 하는지 한 번 더 확인한다.
  - 협업할 때 다른 개발자의 부담을 줄이려면, 내 코드가 받을 수 있는 입력의 범위와 실패 응답을 명확히 만들어야 한다는 점을 명심한다!
