### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - `/posts/1`처럼 리소스를 식별하는 값은 query parameter가 아니라 path variable로 처리하기로 했다.
  - 이를 위해 `/posts/{postId}`처럼 path variable이 들어갈 위치를 미리 정의하고, 내부적으로는 `/posts/[^/]+` 형태의 정규식으로 요청 경로를 매칭하도록 구현했다.
- 그렇게 판단한 이유:
  - `postId`는 게시글이라는 리소스를 식별하는 핵심 값이기 때문에 URL path에 포함되는 것이 의미상 더 적절하다고 판단했다.
  - Query parameter는 `?postId=1`처럼 사용할 수도 있지만, 보통 필터링, 정렬, 검색 조건처럼 부가적인 조건을 표현할 때 더 적합하다고 보았다.
  - 이후 좋아요 증가 API에서도 `/posts/like/{postId}`처럼 동일한 방식이 필요했고, 앞으로 댓글 작성 기능에서도 특정 게시글을 식별해야 하므로 재사용 가능성이 높았다.
- 고려했지만 선택하지 않은 대안:
  - `?postId=1` 형태의 query parameter 방식을 고려했다.
  - 하지만 이 방식은 게시글 자체를 식별한다기보다 요청에 추가 조건을 붙이는 느낌이 강해서, 게시글 상세 조회나 좋아요처럼 특정 리소스에 접근하는 API에는 적합하지 않다고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - path variable 처리 로직을 `Route`에서 공통으로 처리하게 되면서, 특정 endpoint마다 문자열을 비교하거나 정규식을 반복 작성하지 않아도 되었다.
  - `/posts/{postId}`, `/posts/like/{postId}`처럼 여러 API에서 같은 방식으로 동적 경로를 처리할 수 있게 되어 라우팅 규칙이 일관되었다.
  - 앞으로 comment 작성처럼 게시글 id가 필요한 기능을 추가할 때도 기존 path variable 매칭 로직을 재사용할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
    - 가장 최근 게시물을 조회하는 쿼리를 작성하면서 하나의 SQL에 너무 많은 책임이 들어갔다.
    - 게시물 정보뿐만 아니라 이전/다음 게시물 id, 작성자 닉네임, 작성자 프로필 이미지, 게시물 이미지 경로까지 한 번에 조회하려다 보니 쿼리가 복잡해졌다.
    - 쿼리만 봤을 때 어떤 데이터를 어떤 이유로 가져오는지 직관적으로 파악하기 어려웠고, 이후 다른 조회 로직에서 재사용하기도 애매했다.
- 처음 추측한 원인:
    - 게시물 상세 화면에 필요한 모든 데이터를 한 번에 내려주려면 조인을 많이 사용하는 것이 자연스럽다고 생각했다.
    - 쿼리를 여러 개로 나누면 DB 요청이 여러 번 발생하므로, 성능상 하나의 쿼리로 처리하는 것이 더 낫다고 판단했다.
- 원인을 확인하기 위해 시도한 방법:
    - 작성한 쿼리를 기준으로 실제로 어떤 데이터가 필요한지 다시 분리해서 보았다.
    - 게시물 자체 정보, 이전/다음 게시물 id, 사용자 정보, 이미지 경로가 모두 같은 책임인지 검토했다.
    - `/posts`, `/posts/{postId}`처럼 조회 방식이 늘어났을 때 이 쿼리를 그대로 재사용할 수 있는지도 확인했다.
- 실제 원인:
    - 성능을 의식해서 처음부터 모든 데이터를 한 번에 조회하려고 한 것이 문제였다.
    - 게시물 조회 쿼리가 게시물 정보뿐 아니라 사용자 정보, 이미지 정보, 페이지 이동 정보까지 모두 책임지면서 쿼리의 의도가 흐려졌다.
    - 아직 기능이 초기 단계라 성능 최적화보다 가독성, 변경 용이성, 재사용성이 더 중요한 상황이었다.
- 해결 방법:
    - 하나의 큰 조인 쿼리로 모든 정보를 가져오는 방식을 중단했다.
    - 게시물 정보 조회, 이전/다음 게시물 id 조회, 사용자 정보 조회, 이미지 경로 조회를 역할별로 분리했다.
    - `PostRepository`는 게시물 중심의 데이터 조회에 집중하고, 사용자 정보와 프로필 이미지는 `PostService`에서 필요한 시점에 조합하도록 변경했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - 하나의 쿼리가 너무 많은 책임을 가지고 있지는 않은지 먼저 확인한다.
    - 성능 최적화가 정말 필요한 상황인지, 아니면 아직은 가독성과 재사용성이 더 중요한 단계인지 판단한다.
    - 처음부터 큰 조인 쿼리를 작성하기보다, 작게 분리한 뒤 실제 병목이 확인되면 그때 조인을 고려한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - `/posts/{postId}`와 같은 path variable 처리 방식이나 게시물 조회 쿼리 분리처럼 이후 기능에서도 재사용할 수 있는 방향을 선택했다.

- 그 접근이 도움이 된 이유:
  - 처음에는 특정 기능 하나를 위한 코드처럼 보였지만, 이후 좋아요 API나 댓글 기능에서도 같은 라우팅 방식을 재사용할 수 있게 되었다.

- 다음에도 반복하고 싶은 점:
  - 기능을 만들기 전에 URL이 어떤 리소스를 표현하는지 먼저 생각하기.
  - 한 번 작성한 코드가 이후 기능에서도 반복될 가능성이 있는지 확인하기.
  - 쿼리나 Controller 로직이 너무 많은 책임을 갖고 있으면 초기에 분리하기.
  - 구현 중간중간 테스트를 실행해서 변경 사항이 기존 기능을 깨뜨리지 않는지 확인하기.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 커밋을 기능 단위로 바로 나누지 못하고 여러 작업을 한 번에 많이 진행했다.
  - 게시물 조회 쿼리를 처음에 너무 크게 작성해서, 이후에 다시 책임을 분리하는 과정이 필요했다.

- 그렇게 된 원인:
  - 기능이 연결되는 범위가 넓었는데, 초반에 전체 흐름을 작게 쪼개서 커밋 단위로 정리하지 못했다.
  - 성능이나 편의성을 먼저 생각해서 하나의 쿼리나 하나의 흐름에 많은 책임을 넣으려는 시도가 있었다.

- 다음에는 다르게 해볼 구체적인 행동:
  - URL, 요청/응답 형식, 저장 경로처럼 기준이 되는 설계는 구현 전에 먼저 결정하기.
  - 큰 쿼리나 큰 Controller 로직을 작성하기 전에 책임을 나눌 수 있는지 먼저 확인하기.
  - 작업이 30분 이상 길어지면 중간에 커밋 단위가 너무 커지지 않았는지 점검하기.
