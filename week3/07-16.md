# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 프로필 사진 삭제 시 기본 이미지로 다시 표시되도록 수정
  - 프로필 이미지 삭제 기능에 대한 서비스 테스트를 추가
  - 에러 페이지 HTML/CSS를 만들고, 정적 리소스 요청 실패 시 404 에러 페이지로 이동
  - 정적 경로에 맞지 않는 HTTP 메서드 요청 시 405 에러 페이지를 반환하도록 처리
  - 클래스와 public 메서드에 Javadoc 형식 주석을 추가
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 서비스 레이어 분리

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 존재하지 않는 동적/정적 경로 접근 시 에러 페이지를 보여주는 방식으로 404 status를 유지하는 ErrorPageHandler 방식을 선택했다.
- 그렇게 판단한 이유: 존재하지 않는 경로에 대한 요청은 최종 응답이 200 OK가 되어버리면 의미가 왜곡된다. 또한, 크롤러, 모니터링 등 외부 클라이언트에서도 404로 인식되어야 SEO 등에서 정확하다.
- 고려했지만 선택하지 않은 대안: `/error?errorCode=...`로 리다이렉트한 뒤 정적 에러 페이지에서 JS가 `URLSearchParams`로 error code를 읽어 메시지를 매핑하여 보여주는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 에러 판단과 html 렌더링, Response의 책임이 분리되어 각 계층을 독립적으로 테스트하기 쉬워졌다. 에러 페이지 UI 변경이 필요할 시 error/index.html만 수정하면 된다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 존재하지 않는 정적 경로나 동적 경로로 접근했을 때 에러 페이지를 보여주고 싶었지만, 홈으로 redirect하거나 `/error/index.html`로 이동시키면 실제 HTTP 상태 코드가 404가 아니라 200 또는 302처럼 보일 수 있었다.
- 처음 추측한 원인: 에러 페이지를 보여주기만 하면 된다고 생각해서, 에러 메시지를 query parameter로 넘기거나 서버에서 redirect하는 방식으로도 충분하다고 판단했다.
- 원인을 확인하기 위해 시도한 방법: 404, 405 같은 에러의 의미를 다시 알아보고 사용자가 보는 화면과 브라우저/클라이언트가 받는 HTTP 상태 코드가 서로 다를 때 어떤 문제가 생기는지 비교했다. 또한 `StaticResourceHandler`, `Dispatcher`, `RequestHandler` 중 어느 위치에서 에러 응답을 HTML 페이지로 변환하는 것이 자연스러운지도 함께 검토했다.
- 실제 원인: 에러 상황에서 중요한 것은 HTTP 상태 코드를 정확하게 유지하는 것이었다. redirect 방식은 화면 전환은 쉽지만 원래 실패한 요청의 상태를 잃어버릴 수 있고, 정적 페이지 + query parameter 방식은 JS에 의존하게 되어 서버 차원의 일관된 에러 처리가 어렵다.
- 해결 방법: redirect 대신 `ErrorPageHandler`를 통해 404, 405, 500 같은 상태 코드는 유지하면서 HTML 에러 페이지를 반환하는 방향으로 정리했다. 정적 리소스 요청 실패, 허용되지 않은 HTTP Method, `CustomException`, `RuntimeException` 등을 최종적으로 에러 페이지 HTML로 변환하도록 에러 처리 흐름을 구성했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 화면에 어떤 페이지를 보여줄지만 보지 말고 해당 응답의 HTTP status가 실제 상황과 일치하는지 먼저 확인한다. 특히 에러 상황에서는 redirect가 적절한지 아니면 상태 코드를 유지한 채 HTML body만 에러 페이지로 구성해야 하는지 먼저 판단한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 오늘 개발을 진행하면서 현재 코드 구조가 객체지향적으로 적절한지 고민해보았다. 이전에도 특히 책임을 어느 클래스에 둘지, 의존성을 어떻게 관리하는 것이 좋은지 생각하긴 했지만 명확한 기준이 없이 해왔다는 생각이 들어 SOLID 원칙을 다시 학습해보았다.
- 그 접근이 도움이 된 이유: 코드가 변경에 유연한 구조인지 함께 바라볼 수 있었다. 예를 들어 특정 처리 로직을 한 클래스에 생성해서 사용할지 외부에서 주입받도록 할지 고민하면서 단일 책임 원칙이나 개방 폐쇄 원칙이 실제 코드에서 왜 중요한지 이해하는 데 도움이 되었다.
- 다음에도 반복하고 싶은 점: 구현 중에 구조가 어색하다고 느껴지는 부분이 생기면 객체의 책임과 의존성 방향을 먼저 점검해보고 싶다. 또한 SOLID 원칙을 이론으로만 외우는 것이 아니라 실제 코드에서 어떤 원칙과 연결되는지 비교하면서 학습하는 방식을 반복하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: Javadoc 작성 요구사항이 있다는 것을 알고 있었지만, 개발을 진행하면서 클래스와 public 메서드에 바로바로 주석을 작성하지 않고 나중에 한 번에 정리하려고 했다. 그 결과 작성해야 할 양이 너무 많아졌고, 결국 내가 충분히 고민하며 작성하기보다 AI에게 많은 부분을 맡기게 되었다.
- 그렇게 된 원인: 처음에는 Javadoc 작성법을 먼저 학습한 뒤 전체 코드를 한 번에 정리하면 된다고 생각했다. 하지만 실제로는 클래스의 책임, public 메서드의 역할, 파라미터와 반환값의 의미를 각각 다시 파악해야 해서 생각보다 시간이 많이 필요했다. 개발 중에 바로 기록하지 않아 주석 작성이 별도의 큰 작업처럼 쌓이게 되었다.
- 다음에는 다르게 해볼 구체적인 행동: 앞으로는 클래스나 public 메서드를 새로 만들거나 수정할 때 Javadoc과 같은 문서도 그때그때마다 함께 작성한다. 처음부터 완벽하게 쓰려고 하기보다, 최소한 클래스의 책임과 public 메서드의 `@param`, `@return`, `@throws`를 먼저 적어둔다. 이후 리팩터링하거나 문서를 생성하기 전에 표현을 다듬는 방식으로 진행해서, Javadoc 작성이 마지막에 몰리지 않도록 한다.

# 주간 프롬프트 회고 템플릿
> 금요일 데일리 회고 템플릿 아래에 추가해서 작성하세요.

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:

현재 프로젝트에서 존재하지 않는 동적경로와 정적경로로 접근 시 에러 이유와 "/"(홈)으로 이동하는 페이지를 보여주고자 하는데 두 가지 방안을 생각해봤어.

1. error message를 쿼리 파라미터로 넘겨줘서 해당되는 메시지를 javascript를 통해 html에 넘겨서 변경하는 방식
2. 서버 자체에서 기존 html creator처럼 html을 StringBuilder로 만들어서 Response.redirect 되도록 하는 방식

위의 두 가지 방식 중 하나를 선택하고자 하는데 더 나은 방안이 있다면 말해줘. 그리고 각 방안 별로 장단점을 비교해주고 최종적으로 해당 프로젝트를 기준으로 했을 때 선정하면 좋을 방법을 알려줘

선택한 이유: 에러 페이지 어떻게 만들까?라고 묻지 않고 현재 프로젝트에서 고민 중인 두 가지 방안을 먼저 제시했다. 또한 동적 경로와 정적 경로에서 발생하는 문제라는 상황을 설명했고, 각 방식의 장단점 비교와 프로젝트 기준의 최종 선택까지 요구했다. 그래서 AI가 설계 판단 기준까지 포함해서 답변할 수 있었다.

### 2. 원본 프롬프트 구조 분석
- 문제 상황: 존재하지 않는 동적 경로와 정적 경로로 접근했을 때 사용자에게 에러 이유와 홈으로 이동할 수 있는 페이지를 보여주고 싶은 상황이었다.
- 원하는 결과를 어떻게 요구했는가: 내가 생각한 두 가지 방안 중 어떤 방안이 더 나은지 비교해달라고 했고, 더 나은 대안이 있다면 함께 제안해달라고 요청했다. 마지막에는 현재 프로젝트 기준으로 최종 선택하면 좋을 방법까지 요구했다.
- 필요한 맥락이 포함되어 있는가: 정적 경로와 동적 경로 모두에서 발생하는 에러를 처리하고 싶다는 맥락이 포함되어 있다. 또한 기존 프로젝트에서 HTML creator처럼 서버에서 HTML을 생성하는 방식이 있다는 점도 함께 제시했다.
- 판단 기준이나 제약 조건이 있는가: 현재 프로젝트를 기반으로 선택을 해달라는 판단 기준이 있었다. 다만 HTTP status를 유지해야 하는지 redirect를 허용할지 JS 의존성을 줄이고 싶은지 같은 제약은 명확하게 적지 않았다.
- 출력 방식이 명확한가: 각 방안별 장단점 비교와 최종 추천을 요구했기 때문에 답변 형식은 어느 정도 명확했다. 하지만 표 형태로 정리해달라는 식의 구체적인 출력 형식은 없었다.
- 나의 가설, 우려, 의도가 드러나는가: 두 가지 방안을 직접 제시했기 때문에 내가 어떤 선택지 사이에서 고민하는지 드러났다. 또한 더 나은 방안이 있다면 말해달라고 작성해서 내가 제시한 선택지에만 갇히지 않고 더 나은 설계 방향을 찾고 싶다는 의도도 드러났다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
현재 Java로 직접 구현한 WAS 프로젝트에서 존재하지 않는 정적 경로와 동적 경로로 접근했을 때 공통 에러 페이지를 보여주려고 합니다.

현재 고민 중인 방식은 다음과 같습니다.

1. `/error/index.html` 같은 정적 에러 페이지로 이동시키고, errorCode 또는 errorMessage를 query parameter로 넘긴 뒤 JavaScript에서 메시지를 매핑해서 화면을 변경하는 방식
2. 서버에서 기존 HTML creator처럼 에러 페이지 HTML을 StringBuilder로 생성한 뒤 redirect 또는 response body로 내려주는 방식

제가 고민하는 기준은 다음과 같습니다.

- 404, 405 같은 HTTP status code가 실제 응답에 올바르게 유지되는가
- 정적 리소스 요청과 동적 라우팅 요청을 일관되게 처리할 수 있는가
- JavaScript에 의존하지 않고 서버 단에서 에러 처리를 관리할 수 있는가
- 현재 프로젝트 구조의 `StaticResourceHandler`, `Dispatcher`, `RequestHandler`, `Response` 흐름과 잘 어울리는가
- 유지보수할 때 에러 메시지와 에러 페이지 관리가 복잡해지지 않는가

위 기준을 바탕으로 다음 내용을 설명해주세요.

1. 두 방식의 장단점 비교
2. 두 방식보다 더 나은 대안이 있다면 제안
3. 현재 프로젝트 구조에서는 어떤 방식을 선택하는 것이 가장 적절한지
4. 선택한 방식의 이유
5. 구현한다면 어떤 클래스가 어떤 책임을 가져야 하는지
```

### 4. AI가 개선한 프롬프트 분석
- 원본 프롬프트에 비해 문제 상황이 더 구체적이다. 단순히 “에러 페이지를 보여주고 싶다”가 아니라, Java로 구현한 WAS 프로젝트에서 정적 경로와 동적 경로의 에러를 공통으로 처리하려는 상황이라는 점이 명확해졌다.
- 판단 기준이 더 명확해졌다. 원본 프롬프트에서는 현재 프로젝트 기준이라고만 적었지만, 개선안에서는 HTTP status 유지, JS 의존성, 정적/동적 요청의 일관성, 프로젝트 구조와의 적합성, 유지보수성을 기준으로 제시했다.
- 출력 형식이 더 명확해졌다. 장단점 비교, 대안 제시, 최종 선택, 선택 이유, 클래스별 책임까지 요구했기 때문에 답변이 구조적으로 정리될 가능성이 높다.
- 나의 의도와 우려가 더 잘 드러난다. redirect를 쓰면 상태 코드가 바뀔 수 있다는 점, JavaScript에 의존하면 서버 차원의 에러 처리가 힘들어질 수 있다는 점, 현재 핸들러 구조와 맞게 작성되어야 한다는 점이 프롬프트 안에 포함되어 있다.
