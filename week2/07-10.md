# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: `RequestValidater`를 만들어서 request dto에서 사용할 null, empty 검사, 이메일 유효성 검사를 진행할 메서드를 만들었다. 이후에는 회원가입에서 사용하는 request dto에서 id, name, password를 4자 미만이면 예외처리를 하고 email이 유효한 형태가 아닐 경우도 예외처리를 진행하였다. 직무 밋업을 진행하면서, ICT본부 일부 팀에서 하고 있는 업무에 대해 알아볼 수 있었으며 신입 개발자로서 갖추어야 할 역량으로서 '기록'이 중요하다는 점을 인지할 수 있는 유익한 시간이었다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 스쿼드 세션에서 알게된 functional interface, final class 학습, stream 활용법 학습

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 회원가입 요청값 검증을 각 DTO 내부에서 직접 문자열 비교로 처리하지 않고 `RequestValidater`라는 공통 검증 클래스를 만들어 `null`, empty, 이메일 형식 검사를 재사용할 수 있도록 했다.
- 그렇게 판단한 이유: `userId`, `password`, `name`, `email`처럼 request dto에서 반복적으로 검증해야 하는 값들이 있고, 이를 매번 `value == null || value.isEmpty()`처럼 작성하면 중복이 늘어나고 검증 기준이 흩어질 수 있다고 판단했다.
- 고려했지만 선택하지 않은 대안: `UserCreateRequest` 내부에서 모든 검증 로직을 직접 작성하는 방식, 각 필드마다 개별적으로 `null`/empty/길이/이메일 형식을 검사하는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 공통 검증 기준을 한 곳에서 관리할 수 있게 되었고, 이후 다른 request dto에서도 같은 검증 메서드를 재사용할 수 있는 기반이 생겼다. 또한 `RequestValidater` 단위로 테스트를 작성하면 검증 로직 자체를 독립적으로 확인할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `RequestValidater`에 이메일 유효성 검사 메서드를 만들었지만, 실제 회원가입 request dto에서는 이메일 검증에 `isEmail`이 아니라 `isBlank`를 적용해서 의도한 이메일 형식 검사가 수행되지 않았다.
- 처음 추측한 원인: 회원가입 요청 처리 과정에서 예외가 제대로 발생하지 않거나 이메일 검증 정규식이 잘못되었을 수 있다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: `UserCreateRequest`에서 request body 값을 꺼낸 뒤 어떤 검증 메서드를 호출하는지 확인했고, `RequestValidater`에 만들어둔 메서드와 실제 사용 지점이 일치하는지 살펴보았다.
- 실제 원인: 이메일 유효성 검사용 메서드는 만들어두었지만, 실제 `UserCreateRequest`에서 해당 메서드를 호출하지 않고 빈 값 검사만 하고 있었다.
- 해결 방법: 이메일 필드에는 `isBlank`뿐 아니라 `isEmail` 검증이 적용되어야 한다는 점을 확인하여 변경하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 유틸 메서드를 구현한 뒤 실제 호출부에서 해당 메서드를 사용하고 있는지 확인하고 검증 로직은 테스트 코드를 작성하여 정상/비정상 케이스를 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 스쿼드 세션에서 다른 분의 코드에 초기 DDL을 실행해 테이블을 생성하는 흐름이 있었고, 각각의 쿼리를 문자열로 실행하는 구조를 보면서 중복되는 실행 로직을 줄일 수 있지 않을까 하는 개선 방향을 제안해보았다. 또한 이전에 미처 놓쳤던 세션 만료 방식과 쿠키 객체 생성에 대해 스쿼드 세션 팀원들과 공유하였다.
- 그 접근이 도움이 된 이유: 지난 스쿼드 세션에서는 다른 사람의 코드를 듣고 넘어가는 경향이 있었는데 그렇다보니 학습에 도움이 되지 않는다고 판단하여 적극적으로 반복되는 구조나 개선 가능성을 찾아보면서 코드 품질 관점에서 생각해볼 수 있었다. 또한 내가 학습하고 구현하면서 알게 된 세션 만료와 쿠키 객체 생성 내용을 공유하면서 나만의 시행착오를 다른 사람의 학습 자원으로 확장할 수 있었다.
- 다음에도 반복하고 싶은 점: 다른 사람의 코드를 볼 때 동작 여부만 확인하지 않고 중복, 책임 분리, 확장 가능성 관점에서 개선점을 찾아본다. 그리고 내가 겪은 문제와 해결 과정을 혼자 정리하는 데서 끝내지 않고 팀원들과 공유해 함께 학습할 수 있도록 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: `RequestValidater`에서 이메일 유효성을 검사하는 메서드를 작성해두고 나서, 실제 유저 생성 request에서는 이메일 유효성 검사 메서드가 아니라 `isBlank`를 적용하여 원하는 이메일 유효성검사를 진행할 수 없었다.
- 그렇게 된 원인: 테스트 코드를 먼저 작성했으면, 커밋을 날리기 전에 해당 문제를 먼저 파악할 수 있었을 텐데 테스트 코드를 작성하지 않아서 생긴 문제라고 생각한다.
- 다음에는 다르게 해볼 구체적인 행동: 실제 페이지에서 동작하는 과정을 보고 문제점을 찾는 것도 좋지만 그러면 어떠한 과정에서 문제가 생겼는지를 신속하게 파악하기 힘들기 때문에 테스트 코드를 작성하는 것을 꼭 습관으로 만든다.

- # 주간 프롬프트 회고 템플릿

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트: "/index.html"에 접근했을 때, 로그인 상태면 특정 html 조각을 보내주고 index.html에 해당 부분을 삽입. "/index.html"에 접근했을 때 비로그인 상태면 또 다른 특정 html 조각을 보내주고 index.html 해당 부분을 삽입하는 방식으로 만드려면 어떤 구조로 만들어야 하는지 알려줘. 그리고 그러한 방식이 갖는 구조적인 이점에 대해서 논리있게 설명해줘.

선택한 이유: 이 프롬프트는 해결하고 싶은 문제 상황과 원하는 동작이 구체적으로 드러나 있었다. `/index.html` 요청, 로그인/비로그인 상태, HTML 조각 반환, `index.html`에 삽입이라는 흐름을 제시했다. 또한 "어떤 구조로 만들어야 하는지"와 구조적인 이점까지 요구했기 때문에 책임 분리와 설계 판단까지 이끌어낼 수 있었다.

### 2. 원본 프롬프트 구조 분석
- 문제 상황: `/index.html`에 접근했을 때 로그인 여부에 따라 화면의 일부 영역을 다르게 보여주고 싶었다. 전체 HTML을 매번 새로 만들기보다 특정 HTML 조각만 받아서 기존 `index.html`에 삽입하는 구조를 고민하고 있었다.
- 원하는 결과를 어떻게 요구했는가: 로그인 상태일 때와 비로그인 상태일 때 각각 다른 HTML 조각을 반환하고 그 조각을 `index.html`의 특정 위치에 삽입하는 구조를 설명해달라고 요구했다.
- 필요한 맥락이 포함되어 있는가: `/index.html`, 로그인 상태, 비로그인 상태, HTML 조각, 삽입 위치라는 핵심 맥락이 포함되어 있었다. 덕분에 정적 HTML, fragment 요청, 서버에서의 로그인 여부 판단, 클라이언트 흐름을 함께 고려할 수 있었다.
- 판단 기준이나 제약 조건이 있는가: 특정 HTML 조각을 반환하고 삽입하는 방식을 전제로 두었다. 또한 구조적인 이점을 논리적으로 설명해달라고 하여 책임 분리와 유지보수성 관점에서 판단하도록 요구했다.
- 출력 방식이 명확한가: 코드 구현을 요구하지 않고 어떤 구조로 만들지와 그 구조가 갖는 이점을 설명해달라고 했다. 따라서 답변은 설계 흐름, 책임 분리, 장단점 중심으로 정리될 수 있었다.
- 나의 가설, 우려, 의도가 드러나는가: 전체 페이지를 다시 만드는 것보다 변경되는 일부 HTML만 따로 받아 삽입하는 방식이 더 나을 수 있다는 가설이 드러났다. 동시에 이 방식이 구조적으로 타당한지, 책임이 잘 나뉘는지 확인하고 싶은 의도도 명확했다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
`/index.html`은 정적 HTML로 제공되고 있고, 로그인 여부에 따라 header 메뉴 영역만 다르게 보여주고 싶습니다.

구체적으로는 다음과 같은 흐름을 고민하고 있습니다.

- 사용자가 `/index.html`에 접근하면 기본 `index.html`은 그대로 반환한다.
- `index.html` 안의 특정 영역만 비워두고, JavaScript에서 별도 요청을 보낸다.
- 로그인 상태라면 서버가 로그인 사용자용 HTML 조각을 반환한다.
- 비로그인 상태라면 서버가 비로그인 사용자용 HTML 조각을 반환한다.
- 클라이언트는 받은 HTML 조각을 `index.html`의 특정 영역에 삽입한다.

이 방식을 사용하려면 서버와 클라이언트 구조를 어떻게 나누는 것이 좋을까요?

특히 아래 관점에서 설명해주세요.

- `/fragment/auto-menu` 같은 별도 endpoint를 두는 것이 적절한지
- `Controller`, `Service`, `HTML 생성 객체`, `SessionManager`, `CookieUtil`의 책임을 어떻게 나누면 좋은지
- 전체 HTML을 서버에서 생성하는 방식과 비교했을 때 이 방식의 장단점
- JSON으로 로그인 상태만 내려주고 JavaScript에서 DOM을 만드는 방식과 비교했을 때의 장단점
- 현재 같은 WAS 프로젝트에서 이 방식을 선택했을 때 테스트와 유지보수에 어떤 이점이 있는지

코드는 작성하지 말고, 구조와 책임 분리 중심으로 설명해주세요.
```

### 4. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등
- 원본 프롬프트에 비해 현재 상황이 더 명확해졌다. 개선된 프롬프트에서는 `/index.html`이 정적 HTML로 제공되고 있으며, 로그인 여부에 따라 header 메뉴 영역만 바꾸고 싶다는 배경이 먼저 제시되었다.
- 원하는 목표가 더 구체화되었다. `index.html` 반환 → JavaScript의 별도 요청 -> 로그인 상태별 HTML fragment 반환 -> 특정 영역에 삽입이라는 단계별로 정리되었다.
- 비교해야 할 대안이 추가되었다. 전체 HTML을 서버에서 생성하는 방식, HTML fragment를 반환하는 방식, JSON으로 로그인 상태만 내려주고 JavaScript에서 DOM을 구성하는 방식을 비교하도록 요구했기 때문에 더 균형 있는 답변을 받을 수 있다.
- 책임 분리 기준이 명확해졌다. `Controller`, `Service`, `HTML 생성 객체`, `SessionManager`, `CookieUtil`의 역할을 어떻게 나눌지 질문에 포함되면서 객체지향적인 구조 분석을 유도하게 되었다.
- 답변 품질이 좋아질 것으로 예상되는 이유는 개선된 프롬프트가 문제 상황, 원하는 동작, 비교 기준, 제약 조건을 모두 포함하고 있기 때문이다. 따라서 AI가 추측해야 하는 부분이 줄어들고 현재 프로젝트 구조에 맞는 설계 중심 답변을 생성할 가능성이 높아진다.
