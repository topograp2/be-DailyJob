### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 기존에는 Cookie를 생성하는 과정에서 `sid={}; Path="/"`라는 String을 만드는 방식으로 구현하였는데, Cookie 객체를 만들고 빌더패턴으로 생성하도록 변경하였다.
- 그렇게 판단한 이유: 빌더패턴을 적용하게 되면 호출하는 쪽에서 `.httpOnly().secure().maxAge(3600)` 이런 식으로 선택속성을 파라미터 순서에 상관하지 않고 가독성 있게 생성할 수 있기 때문이다.
- 고려했지만 선택하지 않은 대안: sid, path, HttOnly, MaxAge와 같은 쿠키 요소를 파라미터로 받아서, String을 생성하게 하는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 쿠키 생성 규칙이 `Cookie` 객체에 있어서 호출부의 파라미터 순서 실수 가능성이 줄었고, 테스트에서도 name/value 같은 쿠키의 의미 단위를 검증할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 로그인 성공 후에는 세션 쿠키를 발급해야 하고, 로그아웃 시에는 세션을 삭제하면서 브라우저의 쿠키도 제거해야 했다. 이 과정에서 `Cookie`, `SessionStorage`, `UserService`, `UserController` 중 어떤 객체가 어떤 책임을 가져야 하는지 판단하기 어려웠다.
- 처음 추측한 원인: 세션과 쿠키가 모두 로그인 상태와 관련되어 있으므로 `UserService`에서 세션 ID를 생성하거나 쿠키 문자열을 만들고, 로그아웃 시에도 `UserService`에서 관련 처리를 해도 된다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: "로그아웃하려면 쿠키를 삭제시켜야 하는데, sessionId를 찾아서 해당 sessionId를 Map에서 삭제해야 하는 것인지", "쿠키 삭제 시 `Set-Cookie` 헤더를 다시 던져줘야 하는지", "UserService에서 쿠키 문자열을 생성하지 말고 어디서 생성해야 하는지"와 더불어서 그 근거에 대해 알려달라고 AI에게 질문했다. 그 답변을 바탕으로 요청의 `Cookie` 헤더, 응답의 `Set-Cookie` 헤더, 서버 내부의 `SessionStorage`가 각각 어떤 역할을 하는지 현재 코드 흐름과 비교해 확인했다.
- 실제 원인: 쿠키와 세션은 모두 로그인과 관련되어 있지만 책임의 방향이 달랐다. `Cookie` 헤더 읽기, `Set-Cookie` 응답 생성, 서버 세션 저장소 조작, 사용자 인증 로직이 한 객체에 섞일 경우 `UserService`가 HTTP 세부사항까지 알게 되는 문제가 있었다.
- 해결 방법: `UserService`는 사용자 인증만 담당하게 두고, 세션 생성/삭제는 `SessionManager`, 쿠키 생성은 `Cookie`/`CookieUtil`, 응답에 쿠키를 추가하는 일은 `Response`, 전체 흐름은 `UserController`가 맡도록 책임을 나누었다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 특정 로직이 도메인 규칙인지, HTTP 요청/응답 처리인지, 서버 상태 저장소 조작인지 먼저 구분하고, 그 책임을 이미 담당하는 객체가 있는지 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 같은 조원 분께서 `/user/123`과 같은 path variable 처리 구현 방법을 설명해주셨다. 해당 구현에서는 어떤 부분이 path variable인지 구분하기 위해 미리 path variable 위치를 표시해두고 매핑한다고 하셨고, 나는 그 설명을 들은 뒤 내가 이해한 방식이 맞는지 다시 질문하며 확인하였다.
- 그 접근이 도움이 된 이유: 모르는 내용을 그냥 넘기지 않고 재질문하면서, path variable을 식별하고 매핑하는 기준이 무엇인지 더 명확하게 이해할 수 있었다.
- 다음에도 반복하고 싶은 점: 다른 사람의 코드를 보거나 설명을 들을 때 이해가 잘 되지 않는 부분은 넘어가지 않고, 내가 이해한 내용을 다시 질문하고 확인해보면서 확인한다.
