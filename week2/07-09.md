### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 로그인 여부에 따른 페이지 접근 제어를 각 컨트롤러나 서비스에 흩어두지 않고, `AuthFilter`에서 먼저 판단하도록 분리했다.
- 그렇게 판단한 이유: `/login`, `/registration`, `/mypage`처럼 접근 가능 여부가 로그인 상태에 따라 달라지는 요청은 특정 컨트롤러의 비즈니스 로직이라기보다 요청 처리 전에 적용되어야 하는 공통 인증 정책이라고 판단했기 때문이다.
- 고려했지만 선택하지 않은 대안: `UserController`, `AutoMenuService`, `StaticResourceHandler` 각각에서 로그인 여부를 검사하는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 접근 제어 정책이 한 곳에 모여 중복 검사를 줄일 수 있었고, `AuthFilterTest`에서 로그인/비로그인 상태와 path 조합에 따른 redirect 여부를 독립적으로 검증할 수 있게 되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 로그인 여부에 따라 `index.html`의 일부 메뉴 영역만 다르게 보여주고 싶었지만, 서버에서 HTML을 어떤 방식으로 생성하고 전달해야 할지 판단하기 어려웠다.
- 처음 추측한 원인: 로그인 상태에 따라 화면이 달라지므로 서버에서 전체 HTML을 새로 구성해서 반환해야 한다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 전체 HTML을 생성하는 방식, 로그인 여부와 닉네임 같은 데이터만 전달한 뒤 JavaScript로 DOM을 변경하는 방식, 그리고 `StringBuilder`를 사용해 서버에서 HTML 조각을 생성하는 방식을 비교해보았다. 또한 `StringBuilder`의 스레드 안정성 문제가 현재 구조에서 실제 문제가 되는지도 검토했다.
- 실제 원인: 변경이 필요한 부분은 로그인 메뉴 영역뿐이었고, 전체 HTML을 매번 서버에서 생성하면 변경되지 않는 정적 영역까지 불필요하게 다시 구성해야 했다. 반면 데이터만 내려주고 JavaScript에서 모두 생성하는 방식은 이번 단계에서 요구한 `StringBuilder` 기반 HTML 생성 방향과는 맞지 않았다.
- 해결 방법: `/index.html`은 정적 HTML로 유지하고, 로그인 여부에 따라 달라지는 메뉴 영역만 `/fragment/auto-menu` 요청으로 분리했다. 서버에서는 세션을 통해 로그인 여부를 확인한 뒤 `StringBuilder`로 로그인 사용자용 HTML 조각 또는 비로그인 사용자용 HTML 조각을 생성해 반환했다. 클라이언트에서는 `index.js`가 해당 HTML 조각을 받아 `index.html`의 지정된 영역에 삽입하도록 구성했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 변경하려는 화면이 전체 페이지인지 일부 영역인지 먼저 구분하고,  HTML fragment만 서버에서 생성하면 되는지, 아니면 JSON 데이터와 JavaScript DOM 조작이 더 적절한지 비교한다. 또한 `StringBuilder`처럼 스레드 안전하지 않은 객체를 사용할 때는 공유 필드인지, 메서드 내부 지역 변수인지 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 로그인 세션을 구현한 뒤 세션이 계속 유지되기만 하고 만료 정책이 없다는 점을 인지했다. 이후 `SessionData`에 만료 시각을 추가하고, 세션 접근 시 현재 시각과 만료 시각을 비교해 만료된 세션을 제거하는 Lazy 만료 방식으로 보완했다.
- 그 접근이 도움이 된 이유: 세션은 생성하고 저장하는 것만으로는 충분하지 않고 언제까지 유효한지에 대한 정책이 필요하다. 만료 정책을 추가하면서 `SessionManager.hasSession()`이 유효한 세션인지를 판단하도록 의미가 명확해졌다.
- 다음에도 반복하고 싶은 점: 해당 기능이 실제 서버 환경에서 계속 유지될 때 필요한 수명 주기나 예외 상황까지 함께 점검한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 모든 예외 상황을 `CustomException`으로 처리하려고 하면서, 실제로는 예외가 아니라 정상적인 분기여야 하는 상황에도 `CustomException`을 사용하려고 했다. 예를 들어 쿠키가 없거나 세션이 없는 경우는 비로그인 사용자로 처리하면 되는 흐름인데, 처음에는 이를 예외 상황처럼 다루려 했다.
- 그렇게 된 원인: 로그인 실패나 잘못된 요청처럼 명확한 오류 상황을 처리하면서 만든 `CustomException` 방식을, 모든 실패 가능성에 동일하게 적용하려고 했기 때문이다. 즉, 예외 상황과 정상적으로 발생 가능한 상태를 구분하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동: 예외를 만들기 전에 해당 상황이 정말 비정상적인 오류인지, 아니면 비즈니스 흐름상 자연스럽게 발생할 수 있는 상태인지 먼저 구분한다. 쿠키 없음, 세션 없음, 비로그인 상태처럼 예상 가능한 흐름은 `Optional`, `null-safe 처리`, 조건 분기 등으로 표현하고, 요청 형식 오류나 처리 불가능한 상태에만 예외를 사용한다.
