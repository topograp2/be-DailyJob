### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - JWT 기반 로그인/회원가입을 구현하면서 토큰을 어디에 저장할지 논의했다.
    - 세션 방식은 경매 진행 중 세션이 만료되면 사용자가 재로그인해야 하는 불편함이 있어 배제하고, 클라이언트가 독립적으로 유효기간을 판단할 수 있는 JWT 방식을 채택했다.
    - 토큰 저장 위치로 HTTP-only 쿠키와 Authorization 헤더를 비교했다. 쿠키는 만료 시간을 클라이언트가 직접 확인할 수 없고, 헤더는 확인은 가능하지만 XSS에 취약하다는 차이를 정리했다.
    - 최종적으로 JWT + HttpOnlyCookie 방식을 채택하고, 경매방 입장 시점에는 사전 재발급을 수행하고 그 외 상황은 401 기반 재발급으로 처리하기로 결정했다.
  - 엔티티 설계와 관련된 내용을 페어프로그래밍하며 학습했다.
    - `@NoArgsConstructor`가 필요한 이유(JPA가 리플렉션으로 엔티티를 조회할 때 기본 생성자를 요구한다는 점)와 `@NoArgsConstructor(access = AccessLevel.PROTECTED)`로 외부의 무분별한 생성을 막을 수 있다는 점을 확인했다.
    - `User.create`처럼 팩토리 메서드로 객체를 생성하는 방식을 학습했다.
    - Flyway가 checksum을 해싱해 DB에 저장하고 이를 비교해 버전 관리를 한다는 점, 그래서 이미 적용된 sql 파일은 임의로 수정하지 않고 새 파일을 추가하는 방식으로 진행해야 한다는 점을 정리했다.
    - path variable은 기본적으로 `required = true`라는 점, PostgreSQL도 PK에 대해 인덱스를 자동으로 만든다는 점을 확인했다.
  - 카드 정보 상세 조회 API를 구현했다.
  - 회의 시작 전 회의록 AI를 잊지 않고 켠 뒤, 작성된 회의록을 검토했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  -  카드 검색 api 구현

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: JWT 토큰을 Authorization 헤더가 아닌 HTTP-only 쿠키에 저장하기로 했다.
- 그렇게 판단한 이유: JWT를 쓰기로 한 핵심 이유는 클라이언트가 서버에 묻지 않고도 만료 여부를 독립적으로 판단할 수 있다는 점이었고, 이 목적만 보면 Authorization 헤더가 더 적합했다. 하지만 헤더에 담아 클라이언트에서 직접 다루면 XSS로 토큰이 탈취될 위험이 있다는 점이 더 중요하다고 판단했다. 쿠키에 담더라도 액세스 토큰의 만료 시간을 길게 주는 것은 탈취 시 피해를 키우므로 적절하지 않다고 봤다.
- 고려했지만 선택하지 않은 대안: Authorization 헤더 방식을 고려했다. 클라이언트가 토큰의 만료 시간을 디코딩해 확인할 수 있어 경매방 입장 전 미리 갱신하는 흐름을 구현하기 쉽다는 장점이 있었다. 하지만 XSS에 취약해 탈취 위험이 쿠키 방식보다 크다고 판단해 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 쿠키를 선택하면서 클라이언트가 만료 여부를 스스로 판단하지 못하게 되었고, 그 결과 경매방처럼 지연이 치명적인 흐름에서는 401을 기다리지 않고 입장 시점에 사전 재발급을 호출하는 별도 로직이 필요해졌다. 즉 보안을 우선한 대신, JWT 토큰 재발급 api 호출이 더 많이 요구되는 방향으로 결정되었다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 경매는 사용자가 빠르게 참여하고 싶은 긴박한 상황인데, 토큰이 만료된 상태에서 일반적인 401 응답 후 재발급 흐름을 그대로 적용하면 재발급이 끝날 때까지의 지연이 사용자 경험에 문제를 일으킬 수 있었다.
- 처음 추측한 원인: 처음에는 다른 API와 동일하게 401을 받은 뒤 Reissue API를 호출하는 방식으로 통일하면 충분하다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 쿠키 방식을 선택하면서 클라이언트가 만료 여부를 미리 알 수 없다는 점을 다시 짚어보고, 경매방 입장이라는 시나리오가 일반적인 API 호출과 다르게 지연에 민감한 상황인지를 검토했다.
- 실제 원인: 일반적인 401 기반 재발급은 요청 실패 → 재발급 → 재요청의 순서를 거치므로 한 번의 왕복이 추가된다. 대부분의 화면에서는 이 지연이 크게 문제가 되지 않지만, 경매방은 사용자가 즉시 입장해 입찰에 참여하려는 시점이므로 이 지연이 문제로 이어진다는 점이 핵심이었다.
- 해결 방법: 경매방 입장 시점에는 401을 기다리지 않고 사전적으로 Reissue를 수행하고, 그 외의 일반적인 API 호출은 기존대로 401 발생 시 재발급하는 방식으로 나누어 처리하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 인증 정책을 정할 때 모든 API에 동일한 재발급 전략을 적용할 수 있는지, 아니면 지연에 민감한 특정 진입 시나리오가 따로 있는지 먼저 구분해서 확인할 것이다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: `@NoArgsConstructor(access = AccessLevel.PROTECTED)`처럼 이전에는 별생각 없이 붙이던 어노테이션을, 페어프로그래밍 중에 왜 필요한지(JPA가 리플렉션으로 엔티티를 생성할 때 기본 생성자가 필요하다는 점)를 짚고 넘어갔다.
- 그 접근이 도움이 된 이유: 관습적으로 붙여온 어노테이션의 이유를 확인하면서, 앞으로 비슷한 코드를 작성하거나 리뷰할 때 근거를 갖고 판단할 수 있게 되었다. 이런 부분이 페어프로그래밍의 실질적인 효과라고 느꼈다.
- 다음에도 반복하고 싶은 점: 익숙하다는 이유로 그냥 넘기던 코드나 어노테이션이 나올 때마다 페어프로그래밍 상대에게 이유를 묻거나 함께 확인하는 습관을 유지하고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 페어프로그래밍을 진행하면서 navigator와 driver의 역할을 명확하게 구분짓지 않았다. 키보드를 잡지 않은 사람도 코드를 같이 들여다보며 타이핑에 개입하거나, 반대로 driver가 설계 판단까지 혼자 진행해버리는 상황이 섞여서 나타났다.
- 그렇게 된 원인: 시작 전에 두 역할의 책임 범위를 숙지하지 않고 바로 코드를 작성하기 시작했다. 그러다 보니 막히는 부분이 생기면 역할 구분 없이 즉흥적으로 서로 끼어드는 방식으로 진행됐다.
- 다음에는 다르게 해볼 구체적인 행동: 페어프로그래밍을 시작하기 전에 누가 driver이고 누가 navigator인지, 각자의 책임 범위를 짧게라도 먼저 정하고 시작할 것이다. 또한 한쪽에 판단이나 타이핑이 쏠리지 않도록 할 것이다.
