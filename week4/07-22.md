# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 장황한 설명을 줄이고, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 엔티티와 도메인 모델을 분리할지 팀원들과 논의했다.
    - 서비스 계층이 JPA 엔티티에 의존하는 구조가 객체지향 설계 관점에서 어떤 한계를 가지는지 이야기했다.
    - 엔티티가 DB 컬럼 타입에 맞춰지면서 도메인 표현을 자유롭게 쓰기 어려운 점을 확인했다.
    - 도메인과 엔티티를 분리하면 테스트와 Lazy Initialization Exception 대응에는 장점이 있지만, 필드 추가 시 도메인, 엔티티, 변환 로직을 모두 수정해야 하는 비용이 생긴다는 점도 함께 정리했다.
    - 현재 팀이 JPA를 공통으로 사용하기로 한 상황에서는 엔티티 직접 사용이 현실적인 선택일 수 있다고 판단했다.
  - API 명세서를 작성하면서 응답 필드, 공용 API 설계, 페이지네이션 방식, 유비쿼터스 언어를 정리했다.
    - 화면별 API를 계속 추가하는 방식은 기획 변경에 취약하므로 공용 API 중심으로 설계하기로 의견을 맞췄다.
    - 목록 조회 페이지네이션은 전체적으로 커서 기반 방식으로 통일하기로 했다.
    - 회의에서 ERD와 API 명세의 용어 및 상태 값을 다시 정리했다.
    - JWT 토큰은 바디가 아닌 쿠키에 저장하는 방향으로 확정했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 패스워드 암호화 정책을 단방향 해시로 할지, 양방향 암호화로 할지 결정하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 현재 프로젝트에서는 도메인 모델과 JPA 엔티티를 전면 분리하지 않고, 우선 엔티티를 중심으로 구현하는 방향이 적절하다고 판단했다.
- 그렇게 판단한 이유: 도메인과 엔티티를 분리하면 서비스 계층이 영속성 기술에 직접 묶이지 않고, 도메인 객체가 더 자유롭게 행위와 상태를 표현할 수 있다. 하지만 현재 프로젝트는 팀 전체가 JPA를 공통 영속성 기술로 사용하기로 했고, 대부분의 기능이 CRUD와 조회 중심으로 구성되어 있다. 이 상황에서 모든 도메인을 엔티티와 분리하면 설계 이점이 실제 구현 효율을 충분히 상쇄하기 어렵다고 봤다. 필드 하나가 추가될 때마다 도메인, 엔티티, 변환 로직, 테스트 데이터를 함께 수정해야 하므로 팀 내 컨벤션이 복잡해지고 소통 비용도 커질 수 있다.
- 고려했지만 선택하지 않은 대안: 전체 도메인에 대해 도메인 모델과 JPA 엔티티를 완전히 분리하는 방식을 고려했다. 이 방식은 외부 기술이 HTTP, 메시지 브로커, 배치 등으로 다양해질 때 유리하고, 도메인 규칙을 영속성 모델과 분리해서 표현하기 좋다. 또한 컨트롤러에서 응답 객체를 만들 때 Lazy Initialization Exception을 줄이는 데도 도움이 될 수 있다. 하지만 현재는 service 측에서 Response dto를 생성하여 controller에서 보내주는 형태라 트랜잭션 밖에서 지연로딩 상의 객체에 접근할 가능성이 없어서 lazy intialization exception은 고려하지 않아도 된다고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 엔티티 중심 구조를 사용하면 JPA dirty checking을 그대로 활용할 수 있고, 간단한 수정 로직에서 명시적인 `save()` 호출과 변환 로직을 줄일 수 있다. 유지보수 측면에서는 팀원들이 같은 규칙으로 빠르게 구현할 수 있다는 장점이 있다. 다만 입찰처럼 동시성이나 상태 전이가 복잡한 영역은 나중에 별도 도메인 모델 분리를 다시 검토할 필요가 있다. 따라서 지금의 결정은 모든 상황에 대한 결론이 아니라, 현재 프로젝트 범위와 팀 상황을 기준으로 한 우선순위 결정이다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: API 명세를 작성하면서 화면에 의존하는 api를 만들게 되었다. 판매 내역, 정산 예정, 경매 상세, 셀러 대시보드처럼 화면 요구사항이 서로 다르지만 데이터 출처가 겹치는 API가 많았다.
- 처음 추측한 원인: 처음에는 각 화면에서 필요한 필드가 다르므로 화면 단위 API를 만들면 프론트엔드가 사용하기 편할 것이라고 생각했다. 하지만 화면 요구사항이 바뀔 때마다 API가 계속 늘어날 수 있다는 문제가 있었다.
- 원인을 확인하기 위해 시도한 방법: 판매 내역의 상품 목록 조회, 경매 상세 조회, 셀러 대시보드 요약처럼 실제 화면 흐름을 기준으로 필요한 필드를 다시 확인했다. 그리고 각 API가 특정 화면에만 묶여야 하는지, 상태 필터링이나 응답 필드 정리로 공용 API로 만들 수 있는지 비교했다.
- 실제 원인: 문제의 핵심은 API를 화면 구조에 맞출지, 서버 도메인 기준으로 설계할지에 있었다. 화면별 API는 당장 구현하기 쉽지만 기획 변경에 취약하고, 공용 API는 초기 설계 기준을 잘 잡아야 한다. 특히 요약성 API에는 불필요한 필드가 포함되기 쉬워서 응답 필드 최적화가 함께 필요했다.
- 해결 방법: 공용으로 사용할 수 있는 API를 우선 설계하기로 했다. 판매 내역의 상품 목록 조회는 상태 필터링을 활용하고, 경매 상세 조회도 여러 화면에서 재사용 가능한 형태로 정리하기로 했다. 대시보드 요약은 필요한 항목만 다시 추려서 `interestCount`처럼 불필요한 필드는 제거하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: API를 만들기 전에 해당 API가 특정 화면의 일회성 요구인지, 도메인 관점에서 재사용 가능한 조회인지 먼저 확인해야 한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 회의 중 나온 용어와 상태 값을 그냥 넘기지 않고 API, ERD, 도메인 언어 기준으로 다시 맞춰보았다. `product`, `consignment`, `watch`, `interest`, `card master`, `card`, `sold`, `unsold`, `winning`, `passed`, `won`처럼 혼용되던 표현을 하나씩 확인했다.
- 그 접근이 도움이 된 이유: 용어가 흔들리면 API 명세, DB 컬럼, 코드의 enum, 프론트엔드 필드명이 모두 달라질 수 있다. 팀이 같은 개념을 같은 단어로 이해하기 위한 기준을 맞추는 과정이었다. 특히 `product`를 `consignment`로 바꾸고, `card master`를 `card`로 통일한 것은 이후 코드와 문서의 혼란을 줄이는 데 도움이 된다.
- 다음에도 반복하고 싶은 점: 도메인 용어가 어색하거나 중복될 때 즉시 기록하고, ERD와 API 명세까지 함께 수정 범위로 잡는 방식을 유지하고 싶다. 유비쿼터스 언어는 회의에서 말로만 정하면 쉽게 다시 섞이기 때문에, 결정된 표현은 문서와 컬럼명, 엔드포인트에 바로 반영해야 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: API 명세를 작성하면서 필요한 필드와 불필요한 필드를 처음부터 명확하게 구분하지 못했다. 정산 예정, 판매 내역, 셀러 대시보드 요약처럼 요약성 API는 응답 값이 많아질수록 화면 구현은 쉬워질 수 있지만, 서버 입장에서는 API 책임이 흐려질 수 있다. 그런데 회의 초반에는 각 화면에서 당장 필요해 보이는 값을 먼저 넣는 방식으로 생각했다.
- 그렇게 된 원인: API를 도메인 기준으로 설계해야 한다는 점은 알고 있었지만, 실제 명세를 작성할 때는 화면 요구사항을 먼저 따라가게 되었다. 어떤 값이 공용 API에 포함되어야 하는지, 어떤 값은 별도 요약 API나 실시간 조회 API에서 가져와야 하는지 판단 기준이 충분히 정리되지 않았다.
- 다음에는 다르게 해볼 구체적인 행동: 필드를 추가할 때 화면 편의성만 보지 않고 도메인 책임과 조회 비용까지 함께 확인할 것이다.
