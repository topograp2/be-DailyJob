### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: API 응답 DTO를 Java `record`로 정의하기로 했다.
- 그렇게 판단한 이유: 응답 DTO는 주로 데이터를 담아서 클라이언트에 전달하는 역할을 하며, 상태를 변경하거나 복잡한 비즈니스 로직을 가질 필요가 없다고 판단했다. `record`를 사용하면 필드, 생성자, getter를 간결하게 표현할 수 있고, DTO가 불변 객체로 사용된다는 의도도 코드에서 더 명확하게 드러난다.
- 고려했지만 선택하지 않은 대안: 일반 class로 response DTO를 만들고 Lombok의 `@Getter`, `@Builder`, 생성자 annotation 등을 붙이는 방식도 고려할 수 있었다. 이 방식은 익숙하고 확장성이 있지만, 불필요한 코드나 annotation이 늘어날 수 있다고 판단했다. 특히 값 전달 목적의 객체라면 class보다 record가 더 간결하고 목적에 맞다고 봤다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 응답 DTO 코드가 짧아지고 역할이 명확해졌다. 불변 객체로 응답을 다루게 되어 의도치 않은 값 변경 가능성을 줄일 수 있고, 이후 API 응답 DTO가 늘어나더라도 일관된 방식으로 관리할 수 있다. 테스트에서도 응답 값 자체에 집중할 수 있어 DTO 구조를 이해하기 쉬워졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: Swagger 문서를 작성할 때 controller 메서드마다 annotation을 직접 붙이면 코드가 지저분해지는 문제가 있었다. API 명세가 늘어날수록 controller가 실제 요청 처리 코드보다 문서화 annotation으로 더 복잡해질 것 같았다.
- 처음 추측한 원인: Swagger annotation을 controller에 작성하는 방식 자체가 controller의 책임을 늘리고 있다고 생각했다. controller가 요청 처리뿐 아니라 문서화 책임까지 함께 가지면서 가독성이 떨어지는 것이 문제라고 봤다.
- 원인을 확인하기 위해 시도한 방법: 기존에 controller에 Swagger annotation을 붙이는 방식과, 별도 interface에 Swagger 명세를 작성한 뒤 controller가 이를 구현하는 방식을 비교했다. 팀원분이 알려준 interface 분리 방식을 확인하고 Health Check API에 적용해봤다.
- 실제 원인: 문제의 핵심은 Swagger 자체가 아니라, API 문서화 코드와 실제 요청 처리 코드가 한 클래스 안에 섞여 있다는 점이었다. 역할이 분리되지 않아서 controller가 불필요하게 복잡해지고 있었다.
- 해결 방법: `HealthCheckApi` interface를 만들고, Swagger annotation은 해당 interface에 작성했다. 실제 controller인 `HealthCheckController`는 이 interface를 구현하도록 하여 API 명세와 요청 처리 로직을 분리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: annotation이나 설정 코드 때문에 핵심 로직의 가독성이 떨어지고 있지 않은지 먼저 확인할 것이다. controller, service, dto, api spec처럼 각 코드가 어떤 책임을 가져야 하는지 먼저 나누고, 문서화나 설정 성격의 코드는 가능한 분리할 방법을 찾아볼 것이다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 프로젝트 기본 세팅을 하면서 Swagger 명세 분리 방식, response 생성 위치, PK 전략, Flyway 도입 여부처럼 이후 프로젝트 유지보수에 영향을 줄 수 있는 부분을 팀원들과 논의했다.
- 그 접근이 도움이 된 이유: 초기 세팅 단계에서 방향을 잘못 잡으면 이후 기능이 늘어날수록 수정 비용이 커진다. 앞으로 팀원들이 함께 개발하기 좋은 구조인지 고민했다는 점에서 의미가 있었다. 특히 Swagger interface 분리 방식은 controller 가독성을 바로 개선할 수 있었다.
- 다음에도 반복하고 싶은 점: 기능을 구현할 때 지금 당장의 개발 편의성만 보지 않고  이 코드가 나중에 늘어났을 때도 괜찮은 구조인지 확인하는 습관을 유지하고 싶다. 또한 모르는 부분은 팀원과 논의하고, 논의 중 나온 기술적 쟁점은 따로 학습해서 근거를 가지고 결정하는 방식을 반복하고 싶다. 

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: PK 전략 선택에 대해 아직 명확한 기준을 세우지 못했다. auto increment와 UUID의 장단점은 어느 정도 파악했지만, 우리 프로젝트 상황에서는 어떤 선택이 더 적절한지까지 결론을 내리지는 못했다.
- 그렇게 된 원인: 실제 DB 인덱스에서의 효율성, 락 발생처럼 더 깊은 내용까지는 충분히 학습하지 못했기 때문이다. 또한 프로젝트 초기라 실제 데이터 규모나 운영 환경에 대한 가정이 아직 명확하지 않은 점도 있었다.
- 다음에는 다르게 해볼 구체적인 행동: 기술 선택을 할 때 장단점만 정리하는 것에서 끝내지 않고, 우리 프로젝트 기준의 판단 기준을 먼저 세울 것이다.
