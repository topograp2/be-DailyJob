    - 입찰 페이지 진입 시 인증 만료 시간이 얼마 남지 않았을 경우 자동으로 갱신하는 방향을 논의했다.
    - 최종적으로 JWT 기반 인증 방식을 사용하기로 판단했다.

    - 포인트는 차감하지 않고, 현재 최고가로 걸어둔 입찰 금액의 합을 `소멸예정포인트`로 계산해 신규 입찰 가능 여부를 판단하는 방식
    - 최종적으로 후자의 `소멸예정포인트` 방식으로 결정했다.

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 경매 입찰 서비스의 인증 방식으로 JWT 방식을 채택하기로 했다.
- 그렇게 판단한 이유: 입찰 페이지에서는 로그인 만료로 인해 사용자가 입찰을 진행하지 못하는 상황을 최대한 줄여야 한다. JWT는 토큰 자체에 만료 시간이 포함되어 있어 클라이언트가 서버에 매번 요청하지 않아도 만료 임박 여부를 판단할 수 있다. 따라서 입찰 페이지 진입 시 토큰의 남은 시간을 확인하고, 일정 시간 이하로 남았을 때 자동으로 reissue 요청을 보내는 전략을 구현하기에 적합하다고 판단했다.
- 고려했지만 선택하지 않은 대안: 세션 방식도 고려했다. 세션은 서버가 상태를 직접 관리하기 때문에 만료 시간 제어가 명확하고, 금융 서비스처럼 보안이 중요한 서비스에서 많이 사용된다. 또한 상단바에 세션 유지 시간을 표시하거나 입찰 페이지 진입 시 세션을 연장하는 방식도 가능하다. 하지만 세션 만료 시간을 확인하려면 서버 요청이 필요하고, 사용자가 남은 시간을 인지하지 못하거나 시간에 쫓기는 느낌을 받을 수 있다는 점이 있었다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 인증 만료 여부를 클라이언트에서 먼저 판단할 수 있으므로 입찰 페이지의 사용자 경험을 개선할 수 있다. 또한 서버는 reissue 요청을 처리하는 책임에 집중하고, 클라이언트는 토큰 만료 임계치 판단을 담당하는 구조로 역할이 명확해진다. 다만 JWT는 탈취 시 재사용 가능성이 있으므로 access token 만료 시간을 짧게 가져가고, refresh token 관리와 재발급 정책을 명확히 테스트해야 한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 입찰에 참여할 때 포인트를 어떻게 검증하고 관리해야 하는지 결정해야 했다. 여러 경매에 동시에 입찰할 수 있는 상황에서 사용자가 실제 보유 포인트보다 더 많은 금액을 입찰하지 못하게 막아야 했다.
- 처음 추측한 원인: 입찰 시마다 입찰 제시가와 현재 보유 포인트만 확인하면 된다고 생각할 수 있지만, 이미 다른 경매에서 최고 입찰자로 걸어둔 금액이 있다면 그 금액도 함께 고려해야 한다고 판단했다.
- 원인을 확인하기 위해 시도한 방법: 두 가지 방식을 비교했다. 첫 번째는 입찰 시점에 실제 포인트를 즉시 차감하고, 더 높은 입찰자가 나오면 기존 입찰자에게 포인트를 반환하는 방식이다. 두 번째는 실제 포인트를 차감하지 않고, 현재 내가 최고가로 걸어둔 모든 입찰 금액의 합을 `소멸예정포인트`로 계산한 뒤 `소멸예정포인트 + 신규입찰가 > 보유포인트`이면 입찰을 거절하는 방식이다.
- 실제 원인: 핵심은 포인트를 실제로 차감하는 시점보다, 동시에 여러 입찰을 걸었을 때 사용할 수 없는 포인트를 어떻게 계산할지에 있었다. 즉시 차감 방식은 이해하기 쉽지만 입찰 상태가 바뀔 때마다 차감과 반환 처리가 반복된다. 반면 `소멸예정포인트` 방식은 포인트 잔액 자체를 바로 변경하지 않고도 입찰 가능 여부를 판단할 수 있다.
- 해결 방법: `소멸예정포인트` 방식으로 결정했다. 신규 입찰 시 현재 최고 입찰자로 등록된 입찰 금액의 합과 신규 입찰가를 더해 보유 포인트를 초과하는지 검증하는 방향으로 정리했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 돈이나 포인트처럼 수량이 제한된 자원을 다룰 때는 실제 차감 시점, 예약 금액 계산 방식, 동시성 문제가 함께 고려되어야 한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 로그인 유지 방식과 입찰 포인트 처리 방식 등 서비스 정책과 구현 구조가 함께 걸려 있는 주제를 팀원들과 논의했다. 우리 서비스의 입찰 흐름과 운영 방식에 맞는지를 기준으로 비교했다.
- 그 접근이 도움이 된 이유: JWT의 토큰 탈취 위험도 위험하다고 JWT 자체를 배제하여 보지 않고, 우리 서비스가 포인트를 사전 충전한 뒤 입찰에 참여하는 구조라는 점을 함께 고려했다. 이상 거래가 발생하더라도 관리자가 포인트를 조정하거나 롤백할 수 있으므로, 실시간 자금 이동이 발생하는 금융 서비스와 같은 수준의 리스크는 아니라고 판단할 수 있었다.
- 다음에도 반복하고 싶은 점: 기술 선택을 할 때 일반적인 장단점만 비교하지 않고, 우리 서비스의 도메인 구조와 운영 대응 가능성까지 함께 고려하는 방식을 반복하고 싶다. 특히 인증, 포인트, 입찰처럼 장애나 정책 오류가 사용자에게 영향을 주는 부분은 회의에서 충분히 논의한 뒤 근거를 남기고 싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 회의를 많이 했지만 논의 내용을 노션이나 문서에 바로 정리하지 못했다. 그래서 오늘 어떤 논의를 했고, 어떤 근거로 결정했는지가 이전과 비교했을 때보다 명확하게 남지 않았다.
- 그렇게 된 원인: 구두로 계속 이야기하면서 결정은 진행했지만, 회의 중 나온 선택지와 결론을 실시간으로 기록하는 역할이나 습관이 부족했다. 특히 인증 방식과 포인트 처리 방식은 나중에 구현 기준이 되는 중요한 정책인데, 기록이 부족하면 같은 논의를 다시 반복할 가능성이 있다.
- 다음에는 다르게 해볼 구체적인 행동: 회의할 때 논의 주제별로 `문제 상황`, `선택지`, `결정`, `결정 이유`, `추가 확인 필요 사항`을 바로 적어둘 것이다.
