# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:회원가입 시 초기 화면으로 리다이렉트 / query parameter에서 배열을 이용해서, value로 사용하던 것을 Map으로 변경함 
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 중복되는 부분 제거(ENUM etc)

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 쿼리 파라미터를 가져오는 과정에서 기존에는 String []에 value를 사용했는데 Map을 활용하여 저장하게 되었다.
- 그렇게 판단한 이유:코드를 작성하다가 배열로 value 부분만 저장하다보니까 인덱스만 보고 정확히 어떤 값인지 알 수가 없었어서 유저를 저장하는 과정에 닉네임에 이메일을 넣게 되는 등의 문제가 발생했다.
- 고려했지만 선택하지 않은 대안: 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 코드를 작성하면서 해당 값이 무엇을 의미하는지 헷갈리지 않았다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 처음에는 RequestHandler 안에서 HTTP 요청 파싱, URI/query string 추출, 회원가입 처리, 응답 생성까지 모두 메서드로 작성했다. 기능은 동작했지만 코드가 길어져서 내가 쓴 코드를 내가 알아보는 게 힘들었다.
- 원인을 확인하기 위해 시도한 방법: RequestHandler에서 어떤 작업들이 들어가 있는지 역할별로 구분해보았다. 내가 생각하기에 URI를 분석하고 추출하는 것, 유저 관련 동작(회원가입), HTTPRequest 내용 추출이 존재한다고 생각하였다.
- 실제 원인: RequestHandler가 너무 많은 책임을 가지고 있었다. 
- 해결 방법: 역할에 따라 클래스를 분리하고자 했다. 요청 첫 줄을 이용해서 파일 경로와 컨텐츠 타입을 가져오는 로직은  Request로 옮겼고 Query string과 파라미터를 분리하는 로직은 URIParser로 분리했다. 회원가입 관련 비즈니스 로직은 UserService에서 처리하도록 했다. 그렇지만, 응답 관련해서도 분리할 부분이 있어보이고 HTTPRequest의 상태를 바꿀 필요가 있어보인다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 한 클래스 안에 서로 다른 책임이 섞여 있는지 확인한다. 기능을 추가하면서 어느 객체가 책임지는 것이 자연스러운지를 생각해보는 과정을 거친다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 스쿼드 세션에서 `DataOutputStream`과 같은 것을 한 번에 닫을 수 있는 방법을 배우게 되어서 이를 더 알아보고 싶어서 알려주신 분의 코드를 찍었다.
- 그 접근이 도움이 된 이유:오늘 중으로 공부해서 이후에 적용할 수 있어서 도움될 것 같다.
- 다음에도 반복하고 싶은 점:모르는 거 있으면 최대한 기록해야겠다. 다른 사람과의 대화에서 얻는 것이 많은 것 같아 지속적으로 대화를 이어나가야겠다. 또한, 내 코드를 보여주는 것을 꺼려하지 않아야 좋은 피드백을 받을 수 있다고 느꼈다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 처음에 `RequestHandler`에 모든 내용을 담았던 점이 부족했던 것 같다. 그리고 다른 분들의 작업 과정을 보면서 설계하시고 설계를 문서화한 후 코드를 작성하시는 모습을 보고, 그냥 무작정 코드를 작성했던 나의 지난 모습이 아쉽다고 생각했다.
- 그렇게 된 원인: 어떻게 나눠야할지가 막막해서 일단 두었다. 빨리 하고 싶다는 생각이 강하게 들어서 먼저 코드를 작성했다.
- 다음에는 다르게 해볼 구체적인 행동: 주어진 요구사항을 보고 어떻게 구현할지를 먼저 기록한 후 나누던가, 다 작성한 다음에 설계를 한 번 해보는 게 좋을 거 같음.

### 2. 원본 프롬프트 구조 분석
- 문제 상황: Java로 WAS를 구현 중이며 `WebServer.java`에서 `newFixedThreadPool`로 스레드풀을 관리하고 있는데, 더 나은 대안이 있는지 검토하고 싶다.
- 원하는 결과를 어떻게 요구했는가: 각 대안의 장단점, 치명적 약점, 관련 개념, 최종 선택 전략과 그 이유, 추가로 공부할 CS 지식을 서술해달라고 요구했다.
- 필요한 맥락이 포함되어 있는가: Java 기반 WAS 프로젝트이고 `WebServer.java`에서 `newFixedThreadPool`을 사용 중이라는 핵심 맥락은 포함되어 있다.
- 판단 기준이나 제약 조건이 있는가: 코드를 수정하지 말 것, 현재 방식과 비교할 것, 각 방식의 약점과 개념을 설명할 것, 최종 전략을 논리적으로 선택할 것이라는 기준이 있다.
- 출력 방식이 명확한가: 요구사항이 번호로 제시되어 전체 구성은 명확하지만, 표 형식이나 섹션 형식 같은 구체적인 출력 형식은 지정하지 않았다.
- 나의 가설, 우려, 의도가 드러나는가: 현재 `newFixedThreadPool`이 최선인지 의심하고 있으며, 대안들을 비교해 더 적절한 동시성 처리 전략을 선택하려는 의도가 드러난다.

### 4. AI가 개선한 프롬프트 분석
- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등
```
- 원본 프롬프트에 비해 현재 프로젝트의 맥락이 더 구체화되었다. `Java로 직접 WAS를 구현하는 학습용 프로젝트`, `ServerSocket.accept()`, `Socket을 RequestHandler에 전달`, `newFixedThreadPool(100)`처럼 현재 구조가 명시되어 답변자가 상황을 덜 추측해도 된다.
- 목표가 더 명확해졌다. "더 적절한 스레드/동시성 처리 전략을 검토하고 최종 전략을 선택한다"는 방향이 분명해졌다.
- 판단 기준이 추가되었다. 요청이 몰렸을 때의 안정성, 구현 복잡도, 학습 가치, 확장성을 기준으로 비교해달라고 명시해 답변의 평가 기준이 더 일관된다.
- 비교할 대안의 범위가 명확해졌다. `newFixedThreadPool`, `ThreadPoolExecutor + bounded queue`, `newCachedThreadPool`, `Virtual Thread`, `NIO/Event Loop`을 포함하도록 지정해 빠지기 쉬운 대안을 방지했다.
- 출력 형식이 더 구체화되었다. 각 방식마다 `개념 설명`, `나은 점`, `나빠질 수 있는 상황`, `치명적인 약점`, `현재 프로젝트 적합성` 순서로 정리하게 해 답변 구조가 예측 가능해졌다.
- 답변 품질이 좋아질 것으로 예상된다. 답변자가 임의로 대안을 고르거나 추상적으로 설명할 가능성이 줄고, 현재 프로젝트 상황에 맞춰 비교와 결론을 제시할 가능성이 높아진다.
```
