### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 로그인 시, 응답하는 Response에 메서드 체이닝 방식을 이용해서 헤더에 쿠키값 추가하기
- 그렇게 판단한 이유: 아래에 적은 대안으로 코드를 작성하고 있었는데, 그렇게 하면 다른 메서드에서도 계속해서 쿠키를 추가할 때마다 메서드가 계속 늘어나게 되기 때문에 쿠키를 추가하는 로직이 추가된 메서드의 경우 거의 일회용이 될 거 같다고 판단하였기 때문이다.
- 고려했지만 선택하지 않은 대안: redirect(..., String cookieString)과 같이, 오버로딩해서 메서드 하나 더 만들기 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: redirect가 아닌 다른 응답에서도 재사용할 수 있게 되었고, 쿠키를 추가한다는 의미가 더 명확해졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: Dispatcher에서 새로운 컨트롤러가 생길 때마다 private static final UserController userController = new UserController(); 같은 필드를 계속 추가해야 하는 문제가 있었다. 또한 /user/create, /user/login 같은 동적 route가 늘어날 때마다 dynamicRoutes에도 계속 route를 추가해야 했다.
- 처음 추측한 원인: Set<Route> dynamicRoutes로 동적 경로 여부만 판단하고 있었기 때문에, Dispatcher는 해당되는 route를 누구에게 주어야할지 알 수 없었다.
- 원인을 확인하기 위해 시도한 방법: Dispatcher 구조를 보고 책임을 나누어보고자 했다. 
  #### 기존의 Dispatcher가 했던 역할
  - dynamic route 목록을 알고 있음
  - UserController를 직접 생성
  - dynamic route면 UserController로 보냄
  - 그 다음에 내가 한 일 :http method와 path로 구성된 Route가 들어있는 Set<Route> 구조를 검토했다.
- 실제 원인: handler의 역할이 거의 없었다. POST /user/create가 UserController로 가야 한다는 관계가 자료구조 안에 표현되지 않았다. 그래서 컨트롤러가 늘어날수록 Dispatcher가 구체 컨트롤러를 직접 알고, 직접 생성하고, 직접 분기해야 하는 구조가 됐다.
- 해결 방법:  HttpHandler 인터페이스를 만들고, UserController와 StaticResourceHandler가 handle(Request): Response 형태를 따르게 했다. 그 다음 Dispatcher에서 Set<Route> 대신 Map<Route, HttpHandler>를 사용하도록 구조를 바꿨다. Dispatcher는 route로 handler를 찾고, 있으면 handler.handle(request)를 호출한다. 없으면 정적 파일 처리를 위해 StaticResourceHandler로 넘긴다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 어떤 객체가 하고 있는 일에 대해 파악하고, 그 객체와 다른 대상간의 관계를 정리해본다. 특히나, 공통 인터페이스를 설정함에 따라 처리 객체를 같은 타입으로 다룰 수 있게하는 방법은 꼭 기억해두어야겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 구현 도중 잘 알지 못하는데 사용한 메서드와 구현 방식의 경우에는 codex에게 학습 문서를 만들어달라고 요청하여 md 파일로 만들었고 중간에 코드를 짜고 싶지 않을 때 읽었다.
- 그 접근이 도움이 된 이유: 잘 몰랐던 부분을 넘어가지 않고 복습할 수 있어서 기억에 오래 남을 것 같다. 
- 다음에도 반복하고 싶은 점: 코덱스에게 물어본 내용 중에 처음 들어봤거나 생소한 부분은 나중에 찾아볼 수 있게 문서로 만들어달라고 한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: Header가 중복될 수 있음을 이미 알고 있는 상태였는데 알고도 바로 바꾸지 않고, 정말 필요할 때에 바꾸게 되었다.
- 그렇게 된 원인: 헤더 구조를 바꾸면 그밖에 코드도 많이 바꾸어야 돼서 미루게 되었다.
- 다음에는 다르게 해볼 구체적인 행동: 어차피 나중에 해야할 일임을 인지하고 당장 진행하거나 해야할 목록에 추가해야겠다.
