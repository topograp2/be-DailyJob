# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - `Request`에서 기존에 request line만 담았는데 method, http version, path, headers, body를 담도록 구조 변경
  - `URIParser`를 통해 path를 읽어왔던 작업을 `RequestParser`에서 수행하고, method 등도 읽어올 수 있도록 변경
  - `RequestBody`에서 `Content-Length`만큼 body 읽어오기
  - Http 응답 객체를 `Response`에서 구현함 (각 상태 코드에 맞게 정적 팩토리 메서드 구현)
  - 동적/정적 파일 처리 분리하기(정적 파일이 존재하지 않을 경우에는 404 리턴)
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 쓰기 책임 분리하기, `UserService`에서 `Request` 직접 접근 대신 DTO 적용해서 값 꺼내오기

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Request가 직접 HTTP 원문을 파싱하지 않게 하고, RequestParser로 파싱 책임을 분리한 판단
- 그렇게 판단한 이유: 구조 개선의 출발점이 됐기 때문이다. RequestParser가 request line, headers, query string, body를 읽고 해석하게 되면서 Request는 파싱된 요청 정보를 담는 객체에 가까워짐.
- 고려했지만 선택하지 않은 대안: Request 생성자에서 request line, headers, body까지 모두 파싱하게 두는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:파싱 되는 부분이 한 곳에 모여 테스트 지점이 명확해졌고, HTTP 규칙 변경 시 수정 범위가 줄어 유지보수성이 좋아졌다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `RequestParser`에서 바디를 파서하는 것을 구현하니까 정적 파일 응답이 되지 않는 상황이 발생했다.
- 처음 추측한 원인: 바디 파서 어딘가에서 잘못되었다고만 인지하였다.
- 원인을 확인하기 위해 시도한 방법: 콘솔 창에 pipe line이 잘못되었다는 것을 확인하였고 콘솔창에서 나온 잘 못 작성된 줄로 돌아가 다시 검토하는 과정을 거쳤다.
- 실제 원인: 바디가 없는 경우에도 br.readLine을 통해 body를 파서하려고 했고 이로 인해서 클라이언트가 다음 줄의 요청을 보낼 줄 알고 서버가 계속 기다리게 되었다.
- 해결 방법: **Content-Length 헤더에 입력된 것만큼 바디를 읽어온 후에** 읽어온 바디를 Map으로 바꾸어서 Request에 전달하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: Content-Length를 너무 무시하고 있었던 것 같다. 각 헤더별로 가지고 있는 의미가 있음을 기억해야겠다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 작업해야 할 사항들을 우선순위화하여 Todo.md에 기록했고 작업이 끝날 때마다 체크함
- 그 접근이 도움이 된 이유: 어디까지 완료가 되었는지 확인할 수 있었고, 어떻게 진행을 해야할지 대략적인 계획을 마련할 수 있었음. 또한, 작업 예측 시간도 어느정도 알 수 있었음.
- 다음에도 반복하고 싶은 점: 작업을 시작하기 전에 어떤 사항을 오늘 진행할지 계획을 세워보는 것

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 동적 / 정적을 어떻게 구분해야 하는지 생각하지 않고 코덱스에 너무 많이 맡겨버렸다.. 그래서 해당 내용을 이해하지 못한 채로 코드가 완성되었다.
- 그렇게 된 원인: 처음에는 해당 내용을 코덱스에게 구현하지 말고 말로 설명해달라고 했는데 잘 이해가 되지 않았다.
- 다음에는 다르게 해볼 구체적인 행동:
  - 어려움을 겪는 부분은 노션에 정리해서 어떤 부분에 대한 어려움을 겪고 있는지 추후에 확실하게 확인할 수 있도록 하기
  - 최대한 작은 단위로 에이전트를 활용하고 작은 단위의 작업이 끝났을 때마다 작업 흐름이 어떻게 되어가고 있는지 확인하고 다음 작업을 맡기기
