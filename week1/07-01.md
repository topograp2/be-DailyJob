###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: http header에 대해 로그를 작성하고 localhost:8080으로 접속하는 과정에서 로딩이 오래 걸림. 로그는 다 찍혀있는 상황임.
- 처음 추측한 원인:파일 경로가 잘못되어서 파일이 응답되지 않는다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 로그를 추가하는 기능을 작성하기 전의 코드는 잘 작동되어서 이전 코드와 현재 코드를 비교해보았다. http header가 어떤 식으로 작성되어 있는지 확인했다.
- 실제 원인: http 요청 메시지는 헤더와 바디 사이가 빈 줄로 구분되어 헤더의 끝임을 알려준다. 나는 line이 null일때만 break하게 해서 헤더 이후 빈 줄을 만나도 null이 아니므로 계속 다음 줄을 기다리도록 구현하여 html이 응답되지 않는 것이었다. (http/1.1의 경우 keep-alive라서 클라이언트가 요청을 다 보내도 소켓 연결이 끊어지지 않아서 계속 대기 상태로 남아있게 됨.)
- 해결 방법: `while ((line = br.readLine()) != null && !line.isEmpty())`와 같이 line의 길이가 0인 상황일때도 break되도록 조건을 추가하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: http 응답의 구조를 파악하지 못해 이러한 상황이 발생했다고 생각해서 다음에는 데이터의 형태가 어떻게 구성되어 있는지 확인해야겠다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:`byte [] body = Files.readAllBytes(...);`처럼 바로 파일 내용을 불러오면 되는데 초기에 `byte [] body = new byte[1024];`와 같이 선언하고 파일 내용을 불러왔다.
